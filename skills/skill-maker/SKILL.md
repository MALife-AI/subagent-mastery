# Skill Maker

짧은 한 문장이면 충분합니다. 대화를 통해 방향을 잡고, 계획을 세운 뒤, 완성도 높은 스킬을 만들어드립니다.

## When to Use

- 사용자가 새 스킬을 만들고 싶을 때
- "~하는 스킬 만들어줘"처럼 짧은 지시가 들어왔을 때
- 기존 스킬을 참고해서 비슷한 스킬을 만들고 싶을 때

## 핵심 원칙

1. **한 문장이면 충분하다**: 사용자가 아이디어만 던지면, 나머지는 인터뷰로 채운다
2. **만들기 전에 물어본다**: 코드를 쓰기 전에 반드시 방향성을 확인한다
3. **계획을 보여준다**: 바로 만들지 않고, 설계안을 먼저 제시한다
4. **공식 포맷을 따른다**: anthropics/skills 표준 SKILL.md 구조를 준수한다

---

## Phase 1: 의도 포착 (Capture Intent)

사용자의 입력이 아무리 짧아도 시작할 수 있다. 입력에서 다음을 추출한다:

- **무엇**: 스킬이 하는 일
- **언제**: 언제 트리거되어야 하는지
- **누구를 위해**: 대상 사용자

추출할 수 없는 정보는 Phase 2에서 물어본다.

## Phase 2: 방향성 인터뷰 (Direction Interview)

**반드시 실행한다.** 스킬을 작성하기 전에 아래 질문 중 아직 답을 모르는 것만 골라 한 번에 물어본다. 질문은 최소화하되, 핵심은 빠뜨리지 않는다.

### 필수 확인 항목

| 항목 | 질문 예시 |
|------|----------|
| **스킬 이름** | 스킬 이름을 어떻게 지을까요? (예: `my-skill`) |
| **핵심 기능** | 이 스킬이 해야 할 가장 중요한 일 1~2가지는? |
| **트리거 조건** | 어떤 상황에서 이 스킬이 자동으로 활성화되면 좋을까요? |
| **결과물 형태** | 최종 결과물은 어떤 형태인가요? (코드, 문서, 파일, 분석 등) |
| **참고할 것** | 참고하고 싶은 기존 스킬, 문서, 예시가 있나요? |

### 인터뷰 형식

```
🎯 스킬 아이디어를 잘 받았습니다!

이해한 내용:
  "{사용자의 원본 입력}" → {1~2문장으로 해석한 의도}

몇 가지만 더 확인하면 바로 설계에 들어갈 수 있습니다:

1. {질문 1}
2. {질문 2}
3. {질문 3}
   ...

간단히 답해주시면 됩니다. 잘 모르겠는 항목은 "알아서"라고 하셔도 OK!
```

- 질문은 **최대 5개**까지만. 사용자가 지치지 않게 한다
- 사용자가 "알아서", "너한테 맡길게" 등으로 답하면, 합리적인 기본값을 적용한다
- 이미 충분한 정보가 있으면 질문 수를 줄인다

## Phase 3: 설계안 제시 (Planning)

인터뷰 답변을 기반으로 스킬 설계안을 작성하여 **반드시 사용자 확인을 받는다**.

### 설계안 형식

```
📐 스킬 설계안

스킬명: {name}
설명: {한 줄 설명}
트리거: {언제 활성화되는지}

📁 생성할 파일 구조:
  skills/{name}/
  ├── SKILL.md          ← 메인 스킬 정의
  ├── references/       ← 참조 문서 (필요시)
  └── scripts/          ← 자동화 스크립트 (필요시)

📋 SKILL.md 목차:
  1. When to Use — 트리거 조건
  2. {섹션 이름} — {설명}
  3. {섹션 이름} — {설명}
  ...

💡 설계 판단:
  - {왜 이런 구조를 선택했는지 1~2가지 근거}

이대로 만들까요? (Y/수정사항)
```

- 사용자가 수정 요청하면 반영 후 다시 제시
- "Y" 또는 긍정 답변이면 Phase 4로 진행

## Phase 4: 스킬 생성 (Build)

설계안이 승인되면 실제 파일을 생성한다.

### SKILL.md 작성 규칙

**Frontmatter (필수):**

```yaml
---
name: {skill-name}
description: "{트리거 키워드를 포함한 상세 설명. 이 필드가 Claude의 자동 트리거를 결정하므로 구체적으로 작성}"
argument-hint: "{사용 예시}"  # 선택
license: MIT
metadata:
  author: {작성자}
  version: "1.0.0"
---
```

**description 작성 핵심:**
- Claude가 이 스킬을 언제 써야 하는지 판단하는 유일한 근거
- 구체적 액션 키워드를 포함: "Actions: X, Y, Z"
- "Use when..." 패턴으로 트리거 조건 명시

**본문 구조:**
- `## When to Use` — 트리거 조건 (필수, 첫 번째 섹션)
- 이후 섹션은 스킬 성격에 맞게 자유롭게 구성
- 전체 500줄 이내 유지. 넘으면 `references/`로 분리

**작성 스타일:**
- 지시형 문장 사용 ("~한다", "~해라")
- Claude가 따라할 수 있는 구체적 절차 제시
- 테이블, 코드블록, 리스트 적극 활용
- 모호한 표현 지양 ("적절히", "필요에 따라" → 구체 기준 제시)

### 파일 생성 순서

1. `skills/{name}/SKILL.md` 생성
2. 필요시 `skills/{name}/references/` 하위 참조 문서 생성
3. 필요시 `skills/{name}/scripts/` 하위 스크립트 생성

## Phase 5: 완료 보고 (Wrap-up)

생성 완료 후 간결하게 결과를 보고한다:

```
✅ 스킬 생성 완료

📁 생성된 파일:
  - skills/{name}/SKILL.md
  - {추가 파일들}

🚀 사용 방법:
  - 슬래시 커맨드: /{name}
  - 또는 자연어로: "{트리거 예시}"

💡 팁: smart-skill 커맨드에 이 스킬을 추가하면
   자동 매칭도 지원됩니다.
```

## 엣지 케이스 처리

| 상황 | 대응 |
|------|------|
| 입력이 매우 모호함 ("스킬 하나 만들어") | Phase 2에서 "어떤 종류의 작업을 도와주는 스킬인가요?"부터 시작 |
| 기존 스킬과 중복 | 중복 가능성을 알리고, 새로 만들지/기존을 확장할지 확인 |
| 너무 복잡한 요구 | 핵심 기능만 v1.0으로 먼저 만들고, 확장은 이후 버전으로 제안 |
| 사용자가 Phase 2~3을 건너뛰길 원함 | 합리적 기본값으로 바로 Phase 4 진행. 단, "기본값으로 진행합니다" 안내 |

## 참고: 좋은 스킬의 특징

- **description이 구체적**: 자동 트리거가 정확하게 작동
- **When to Use가 명확**: 사용자도 언제 쓰는지 바로 이해
- **절차가 단계별**: Claude가 따라하기 쉬운 numbered steps
- **예시가 포함**: 입력/출력 예시로 기대치를 명확히 설정
- **500줄 이내**: 핵심만 담고 상세 내용은 references로 분리
